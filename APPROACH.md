# Approach

A document Q&A assistant for commercial real-estate lawyers, built on the provided
React + FastAPI + Postgres baseline. I focused on the two graded features: answering
questions that span a bundle of documents, and making every answer verifiable back to
the exact page and quote.

## How I architected it, and why

The whole thing is **one agent in a tool loop** (PydanticAI). I deliberately avoided a
multi-agent setup or a graph framework. For a chat assistant, a single agent that decides
which tool to call next is easier to reason about, easier to test, and more than capable
enough. The agent never sees a document until it asks for it; nothing is preloaded into
the prompt, so the bundle size doesn't blow up the context.

It has four tools, each doing one job:

- `list_documents()` — the map. Each document comes with a short routing "card" (kind +
  summary) generated by a cheap model at upload time.
- `grep(pattern, document_id?)` — a regex search over the bundle. The agent writes its own
  pattern and refines it if the first try misses. It runs the pattern as two regex passes (why,
  below).
- `read_pages(document_id, start, end?)` — reads a page range. This is the **only** text the
  agent is allowed to quote and cite.
- `summarize_documents(question, document_ids)` — the breadth path (below).

I went **keyword-first, not embeddings**. Grep is transparent, precise, needs no extra
infrastructure, and a good model is smart enough to search the way a lawyer actually looks
for a clause. It's also the token-efficient choice: the agent pulls only the lines and pages
it needs into context instead of stuffing whole documents in, which keeps each turn cheap and
pairs nicely with Anthropic's built-in conversation compaction on long sessions. There's
actually a growing body of work pointing the same way for agentic search — papers like *"Is
Grep All You Need?"* find that plain grep often matches or beats vector search inside real
agent loops. 

**Why grep runs the regex twice.** First, the pattern runs as a Postgres `~*` (case-insensitive
regex) query, which scans the whole bundle inside the database and returns only the pages that
contain a match — the database does the heavy scanning, and I never pull every page's text into
the app. But a page-level match tells me *which* pages hit, not *which lines*. So a second pass
with Python's `re` runs over only those already-filtered pages, splits each into lines, and keeps
just the matching ones, tagged with their document and page.

**Grounding is the part I refused to compromise on.** The model returns an answer as
markdown plus a list of `{document_id, page, quote}` citations. Before anything reaches the
user, every quote is matched back against the real page text. If a quote isn't actually
there, it's dropped — the model can't invent a citation. Surviving quotes get bounding
boxes computed server-side (PyMuPDF) so the frontend highlights the exact text on the PDF,
and the inline `[n]` markers are renumbered so they always line up. "Not specified" beats a
guess, every time.

### Cross-document, from 3 to 50

Two paths, and the agent chooses:

- **Targeted** ("what's the rent today?") stays in the loop: grep to find the pages, read
  them, cite. This is the common 1–3 document case.
- **Breadth** ("summarise the bundle", "which leases grant parking?") fans out.
  `summarize_documents` reads each document *in its own isolated context, in parallel*
  (cheap Haiku model, capped concurrency) and returns a short summary plus verified quotes
  per document. The main agent then synthesises from those findings — it *is* the reduce
  step. So a 50-document bundle never lands in one context window; only the distilled
  findings do.

## What I prioritised, and what's next

I spent the time on the spine; agentic answers, verified citations, cross-document; and
the things that make it feel real rather than a demo: graceful handling of empty bundles
and off-topic questions, and a real test and eval setup.

**Building the eval harness first.** With an LLM
system you can't one-shot correctness: a prompt tweak that fixes one question can quietly break
three others, and you won't know unless you measure. So I'd say that the eval harness is the first
thing worth having; once it's in place, every prompt edit and model swap becomes a measured
change. So, in future, the first thing I would do is to build an eval suite

Further, I also streamed the agent's **tool calls** live: each time it greps, reads a page, or kicks
off a breadth summary, that step is pushed to the UI as it happens, so you watch it work
instead of staring at a spinner. Getting this end-to-end trace working early mattered — it's
the harness everything else hangs off, and it makes the agent's behaviour legible while
you're building.

With more time, roughly in order:

1. **Building the Evals Harness**  
    I would create a small gold set of realistic lawyer questions with expected answers, source documents, page numbers, and supporting quotes. These would run through the full evals pipeline — retrieval, page reads, answer generation, citation verification, and final response — so I can quantify the performance of architecture, to make directed improvements.
2. **A vector store for retrieval** (embeddings + pgvector), alongside grep. When the query
   doesn't share vocabulary with the document, semantic search fetches the relevant pages
   more accurately than a regex can; the strongest version is hybrid (keyword + vector) with
   a reranker on top, which is the proper precision fix near the top of the 50-document range.
4. **Stream the model's actual reasoning, not just its tool calls.** A pattern I like for
   agentic systems is to get the end-to-end harness working first, then layer capability onto
   it. The trace streams tool calls today; the natural next addition is streaming the model's
   own reasoning (its thinking tokens) so the user sees *why* it takes each step, not just what.
5. The extractive-quote fix described below.

## Interesting problems

**Loose matching of citations.** Currently citations are matched based on regex, which I believe is quite a loose matching and has a high frequency of failures. This can lead to incorrect or missing citations in the final output.

**Keeping it agentic instead of patching edge cases.** My first instinct for an empty bundle
was `if not docs: return`. Instead I made the tools self-describing — the empty-bundle result
tells the agent what to do next — and put an intent taxonomy in the prompt. That generalises:
the same approach handles off-topic questions, greetings, ambiguous references, and documents
that aren't in the bundle, instead of one bypass per case. A related subtlety: the current
document list is injected fresh every turn rather than read from replayed history, so a
document uploaded mid-conversation doesn't quietly become invisible.

## Models and dependencies

Only the provided Anthropic API. For the agent loop I used **Sonnet 4.6** as the starting
point; it's the best balance of speed and intelligence for a tool-using loop. For the basic
summarisation work (document cards, the breadth map) I used **Haiku 4.5**, which is very fast
and plenty for that job. These are sensible defaults rather than tuned choices: once the eval
suite is producing numbers, both the model picks and the prompts are things I'd tune against
real results.
